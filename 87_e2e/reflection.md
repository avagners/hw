# E2E ваше всё

Тестировал веб-приложение для управления подписками на уведомления. Проект представляет собой React-фронтенд.

Для тестов был выбран Playwright.

## Созданны тесты (18 тестовых сценариев)

| Тест | Файл | Сценарии |
| --- | --- | --- |
| #1: Основной сценарий | main-user-flow.spec.ts | 3 теста: полный путь пользователя (логин → каталог → заказ → подписки), навигация, состояния загрузки |
| #2: Управление шаблонами | template-management.spec.ts | 4 теста: создание подписки с шаблоном, просмотр шаблона, смена типа шаблона, валидация |
| #3: Краевые случаи | edge-cases.spec.ts | 11 тестов: 404, пустые состояния, навигация, ошибки, валидация форм, специальные символы, адаптивность |

## Получен эффект

1.  Автоматизация критических путей - основные пользовательские сценарии теперь покрыты тестами
2.  Раннее обнаружение ошибок - тесты находят проблемы интеграции между компонентами
3.  Регрессионная защита - при изменениях в коде тесты покажут, не сломался ли существующий функционал
4.  Документация через тесты - тесты описывают, как система должна работать

## Выводы 

Для меня это первый такой опыт E2E тестов. 
Обычно все пользователиские сценарии тестировались вручную.

Следуя материалу, E2E тесты заложены в фундамент тестирования. Это позволяет обнаруживать интеграционные проблемы, которые возникают между компонентами системы, а не только логические ошибки в функциях.

После выполненного задания я понимаю, почему E2E тесты должны быть в фундаменте. Юнит-тесты проверяют, что код работает правильно. E2E тесты проверяют, что продукт работает для пользователя и пользовательские сценарии выполняются успешно. Это совсем другая философия.
